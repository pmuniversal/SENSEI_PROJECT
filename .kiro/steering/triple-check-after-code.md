# Triple-Check Protocol — Обязательная тройная проверка после КАЖДОГО изменения кода

> Добавлено: 2026-06-17. ОБЯЗАТЕЛЬНО. Нарушать нельзя.

## Правило

**После КАЖДОГО создания, редактирования или удаления ЛЮБОГО кода, функции или логики** —
обязательно пройти тройную проверку:

### 1. 🧪 QA-ТЕСТИРОВЩИК (Мега-тестировщик)
**Что делает:** Протыкивает ВСЕ функции, проверяет ВСЕ сценарии использования.
**ГЛАВНОЕ ПРАВИЛО: Проверять РЕАЛЬНОЕ поведение — читать код и сверять с БД-схемой.**

Чеклист:
- [ ] **СВЕРКА С БД:** Все имена полей в запросах (select, insert, update) СОВПАДАЮТ со schema.sql. Если в коде `profiles.select('current_store_id')` а в БД поле `store_id` — это БАГ.
- [ ] **СВЕРКА С RPC:** Все параметры вызовов rpc() совпадают с functions.sql (имена, типы, nullable).
- [ ] **ПОЛНЫЙ FLOW:** Пройти ВЕСЬ путь пользователя от входа до результата. Не «кнопка рендерится» а «кнопка кликается → данные идут в БД → результат виден на экране → при перезагрузке данные сохранены».
- [ ] Позитивный сценарий (happy path) работает ОТ НАЧАЛА ДО КОНЦА
- [ ] Негативные сценарии (пустые поля, неверный ввод, превышение лимитов)
- [ ] Граничные случаи (0, -1, MAX, пустой список, один элемент)
- [ ] **ПОСЛЕ ДЕЙСТВИЯ:** Что происходит ПОСЛЕ нажатия кнопки? Есть ли redirect/reload/callback? Не зависает ли на loading?
- [ ] UI: кнопки кликабельны, поля принимают ввод, модалы открываются/закрываются
- [ ] Мобильный вид: не ломается на узком экране
- [ ] Десктопный вид: не ломается на широком экране
- [ ] Переключение языков (RU/UZ) — все строки переведены
- [ ] Light/Dark тема — всё читаемо в обоих режимах
- [ ] Нет console.error, нет белого экрана, нет бесконечного loading
- [ ] Данные сохраняются в БД и отображаются после перезагрузки
- [ ] **STATE TRANSITIONS:** После каждого действия (создание, удаление, редактирование) — проверить что UI обновился. Нет ли зависания в промежуточном состоянии (loading без конца, модал не закрылся, список не обновился).

### 2. 📊 БИЗНЕС-АНАЛИТИК (Business Auditor)
**Что делает:** Проверяет бизнес-логику и пользовательский путь.

Чеклист:
- [ ] Пользовательский путь (user flow) логичен и не имеет тупиков
- [ ] Нет путей где пользователь «застрянет» без возможности продолжить
- [ ] Бизнес-правила корректны (цены, скидки, остатки, долги)
- [ ] Данные согласованы (сумма продажи = сумма позиций, остаток уменьшается правильно)
- [ ] Нет путей потери данных (продажа без записи, долг без учёта)
- [ ] Соответствует потребностям МПМ (просто, понятно, на узбекском/русском)
- [ ] Не повторяет слабости конкурентов (сложность, непредсказуемость)

### 3. 🏗️ PRINCIPAL ENGINEER (Технический аудитор)
**Что делает:** Проверяет качество кода и безопасность.

Чеклист:
- [ ] Нет хардкода секретов (ключи, пароли в коде)
- [ ] Ошибки обрабатываются (try/catch, показ ошибки пользователю)
- [ ] Нет утечек памяти (бесконечные интервалы, непрочищенные подписки)
- [ ] SQL-инъекции невозможны (используем RPC, параметризованные запросы)
- [ ] RLS работает (пользователь видит только свои данные)
- [ ] Нет дублирования кода (вынести в компоненты/функции)
- [ ] Код читаем и поддерживаем (именование, структура)
- [ ] Производительность: нет лишних перерисовок, нет N+1 запросов
- [ ] **ИЗОЛЯЦИЯ ДАННЫХ (обязательно перед каждым деплоем на prod):**
  Явно проверить что магазин A не видит данные магазина B.
  Сценарий: войти под тестовым аккаунтом магазина A → открыть товары/долги/продажи →
  убедиться что данные магазина B не появляются в списках, запросах, ИИ-ответах.
  Не «наверное RLS работает», а «проверено руками на 2 тестовых аккаунтах».

## Формат отчёта (обязательный)

После каждого изменения кода агент выводит:

```
═══ TRIPLE-CHECK REPORT ═══

🧪 QA-ТЕСТИРОВЩИК:
✅ [что работает]
❌ [что сломано / не работает] → [как починить]

📊 БИЗНЕС-АНАЛИТИК:
✅ [логика корректна]
❌ [логические ошибки / тупики] → [как исправить]

🏗️ PRINCIPAL ENGINEER:
✅ [код чистый]
❌ [проблемы кода] → [как исправить]

ИТОГО: [X] ✅ пройдено / [Y] ❌ требует исправления
РЕШЕНИЕ: [ДЕПЛОЙ OK / НУЖНЫ ПРАВКИ]
```

## Если найдены ошибки

1. Исправить ВСЕ найденные проблемы
2. Повторить билд + деплой в preprod
3. Повторить Triple-Check
4. Только после «ДЕПЛОЙ OK» — переходить к следующему эпику

## Когда применяется

- После написания нового компонента/функции
- После редактирования существующего кода
- После удаления кода (проверить что ничего не сломалось)
- После любого `vite build` + деплой
- Перед переходом к следующему эпику

## Исключения

НЕ нужно для:
- Изменения документации (.md файлы)
- Изменения конфигурации (.kiro/steering/)
- Чисто текстовые изменения (переводы, комментарии)
