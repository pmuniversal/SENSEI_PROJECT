# Harness-мышление — обвязка важнее модели

> Источник: _PM/RULES/harness-mindset.md
> Основано на исследованиях Anthropic, Stanford, OpenAI Engineering Blog (2026).

## Главный принцип

**Модель — это лошадь. Harness — упряжка вокруг неё.**
Одна и та же модель в разном harness: 6,7% → 68% решённых задач (эксперимент Яна Балюка, 2026).
Промт-подбор даёт <3% прироста. Улучшение обвязки — десятки процентов.

## Правило №1 — Чини harness, не промт

**Нашёл ошибку агента → чини окружение, не слова.**

Вместо: «добавлю в промт "думай шаг за шагом"»
Делай: «добавлю правило в steering / хук / инструмент / проверку»

Это принцип Митчела Хашимота (HashiCorp), февраль 2026:
> «Каждый раз, когда находишь ошибку агента — трать время чтобы починить окружение
> так, чтобы агент эту ошибку больше никогда не сделал.»

Практически для этого проекта:
- Агент сделал ошибку в коде → добавить хук / правило в triple-check / уточнение в engineering-guidelines
- Агент не обновил PROJECT_STATUS → ужесточить project-status-protocol
- Агент забыл записать ошибку → обновить never-repeat-errors

## Правило №2 — Context Rot: перезапускай, не сжимай

**Длинная сессия → контекст заполняется → агент начинает врать.**

Симптомы context rot:
- Агент пишет «готово», хотя задача не сделана
- Агент «забывает» инструкцию из начала сессии (цвет поменялся, требование пропало)
- Агент упрощает задачи без предупреждения («ну и так сойдёт»)
- Бесконечный цикл без прогресса

**Протокол при признаках context rot:**
1. Не пытаться «напомнить» агенту в том же чате — это не поможет
2. Сохранить прогресс → обновить CHECKPOINT_LATEST.md
3. Открыть новую сессию с чистым контекстом
4. Передать агенту: CHECKPOINT_LATEST + конкретную следующую задачу (одну, не список)

**Превентивно:** разбивать длинные задачи на сессии по одному этапу. Принцип «один чекпоинт — одна сессия».

## Правило №3 — Bash-first, MCP по необходимости

**MCP-серверы тратят тысячи токенов описаний при каждом запросе** — ещё до того как ты написал задачу.

Правило инженеров Anthropic из их собственных репозиториев:
- **Bash (терминал)** — для всего рутинного: git, файлы, поиск, базы данных, GitHub
- **MCP** — только для того, без чего никак: OAuth-авторизация, типизированные интеграции, чувствительные данные

Если подключено 5 MCP-серверов — это несколько тысяч токенов в каждом запросе «бесплатно». Контекст засоряется раньше.

## Harness = 7 систем (держать в голове)

1. **Agent Loop** — цикл: модель → действие → результат → снова модель
2. **Tools** — bash, файлы, поиск, веб
3. **Subagents** — параллельные подзадачи с отдельными чистыми контекстами
4. **Hooks/Skills** — автопроверки между шагами (линтер, тесты)
5. **Memory file** — CLAUDE.md / долгая память проекта (steering-файлы)
6. **MCP-серверы** — подключение к внешним системам (по необходимости)
7. **Permissions** — что агенту разрешено трогать

Steering-файлы этого проекта — это и есть harness. Они формируют окружение агента.

## 6 паттернов для сложных задач

Когда задача большая — выбери паттерн:

| # | Паттерн | Когда использовать |
|---|---------|-------------------|
| 1 | **Classify and Act** | Поток разнородных входов (письма, заявки, задачи) |
| 2 | **Fan-out + Synthesize** | Параллельная проверка нескольких аспектов |
| 3 | **Adversarial Verification** | Критическая задача: один пишет, другой ищет дыры |
| 4 | **Generate + Filter** | Нужно много вариантов → выбрать лучший |
| 5 | **Tournament** | Несколько подходов → судья выбирает победителя |
| 6 | **Loop Until Done** | Итеративная задача с чёткими критериями готовности |

## Dynamic Workflows (Claude Code 2.1.154+)

Если используешь Claude Code — три способа запустить сотни параллельных субагентов:
1. Написать `ultracode` перед запросом — разово
2. Сказать «запусти workflow» своими словами
3. `/effort ultracode` — включить на всю сессию (+ максимальный режим рассуждения)

Это полезно для: рефакторинга больших кодовых баз, проверки уязвимостей, миграций.

## Предупреждение про мультиагентность

5 агентов × 95% надёжность = **77% точности** (0,95⁵ — компаундинг ошибок).
Для 90% бизнес-задач **один грамотный агент с хорошими инструментами лучше пяти**.
Мультиагентность — только если задача реально требует разных независимых «голов»:
симуляция переговоров, независимые исследовательские перспективы, мозговой штурм с ролями.
