# Доказательность — работа только на фактах

> Подгружается автоматически. Источник: _PM/RULES/evidence-based.md.
> Правило для всех проектов: предлагать и делать — только на основе
> доказательств и проверенных фактов.

## Главное правило одной строкой

**Нет доказательства — нет утверждения.** Любое решение, рекомендация или вывод
опирается на проверяемый факт, а не на догадку, память или «обычно так».

## Что считается доказательством (по убыванию надёжности)

1. **Прямая проверка в системе** — прочитанный файл, вывод команды, результат
   теста, реальное состояние репозитория/БД/конфига.
2. **Официальная документация** — docs продукта/библиотеки, спецификация, API-референс.
3. **Авторитетный первоисточник** — репозиторий мейнтейнера, RFC, стандарт.
4. **Воспроизводимый эксперимент** — измерение «до/после», лог, скриншот, цифра.

Слухи, «кажется», «вроде работало», ответ по памяти без проверки — **не доказательство**.

## Обязательный протокол перед утверждением

1. **Проверь факт у источника** — прочитай файл / выполни команду / открой доку.
2. **Раздели:** что я **проверил** ↔ что я **предполагаю**. Предположение помечать прямо.
3. **Сошлись на источник** — назвать файл, команду, ссылку, цифру.
4. Если проверить нельзя — **сказать прямо** «не проверено», а не выдавать за факт.

## Как оформлять ответ

- Утверждение о коде/системе → сослаться на конкретный файл/строку/вывод команды.
- Утверждение о технологии/версии/цене → сослаться на доку или свежий источник.
- Бизнес-вывод → опереться на цифру/метрику, а не на ощущение.
- Нет данных → «недостаточно данных, нужно проверить X», без выдумывания.

## Связь с другими правилами

- **principles.md** — измеримость «до/после», champion/challenger: эксперимент
  доказывает превосходство цифрой, а не мнением.
- **never-repeat-errors.md** — необоснованное предположение, оказавшееся неверным,
  записывается в ERRORS_LOG.md как ошибка категории `[ЛОГИКА]`.
- **working-rules.md** — «уточнять факты у владельца, не додумывать».

## Запрещено

- Выдавать предположение за установленный факт.
- Делать вывод о коде/системе, не прочитав код / не выполнив проверку.
- Рекомендовать решение «по памяти», когда есть способ проверить.
- Скрывать неопределённость — если не уверен, это надо сказать.
