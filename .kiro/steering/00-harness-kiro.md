# 00 — Harness-мышление для Kiro-агента

> ⭐⭐ ВЫСШИЙ ПРИОРИТЕТ. Загружается первым (префикс 00).
> Это ядро поведения агента. Прочитай и применяй на каждом шаге.

## Главный принцип

**Модель — это лошадь. Harness (обвязка) — упряжка вокруг неё.**
Одна и та же модель в правильном harness даёт результат в 10 раз лучше.
Ключевое правило: **нашёл ошибку → чини steering-файл (окружение), не промт.**

## Правило 1 — Чини окружение, не слова

Если агент делает ошибку — не добавляй «думай шаг за шагом» в промт.
Вместо этого: добавь правило в steering / уточни инструкцию / улучши структуру задачи.

Примеры:
- Агент не обновил PROJECT_STATUS.md → ужесточить project-status-protocol.md
- Агент забыл записать ошибку → обновить never-repeat-errors.md
- Агент дал неточный ответ → добавить конкретное правило в steering

## Правило 2 — Один диалог = одна задача

**Длинная сессия → контекст переполняется → агент начинает врать.**

Симптомы переполнения контекста (Context Rot):
- Пишет «готово», хотя задача не выполнена
- «Забывает» инструкцию из начала диалога
- Упрощает без предупреждения

Действие при симптомах:
1. Сохранить прогресс → обновить CHECKPOINT_LATEST.md
2. Открыть **новый чат** с чистым контекстом
3. Передать: содержимое CHECKPOINT_LATEST.md + одну конкретную задачу

Превентивно: один этап = один чат = один чекпоинт.

## Правило 3 — Bash-first, MCP по необходимости

Каждый подключённый MCP-сервер добавляет токены в контекст при каждом запросе.
- **Терминал/bash** → для рутинного: git, файлы, поиск
- **MCP** → только когда без него никак (OAuth, типизированные интеграции)

Лайфхак: если MCP нужен редко → попроси агента «создай skill из этого MCP» → получишь лёгкую версию без постоянной нагрузки.

## Правило 4 — Steering-файл = системная инструкция агента

Этот файл и все файлы в `.kiro/steering/` — это и есть harness.
Они читаются при каждом запросе и формируют поведение агента.

Хороший steering: кратко, конкретно, без воды.
Плохой steering: длинный, общий, противоречивый — засоряет контекст.

## Правило 5 — Выбор модели по задаче

| Задача | Модель |
|--------|--------|
| Глубокий ресёрч, архитектура, стратегия | Claude Opus |
| Написание кода, исполнение по инструкции | Claude Sonnet |
| Быстрые простые задачи, массовые запросы | Claude Haiku |

Thinking Mode — держать ВКЛЮЧЁННЫМ всегда. Без него качество резко падает.

## Правило 6 — Скилы: агент выбирает сам

Агент читает только `name` + `description` из каждого скила — не весь файл.
Это экономит контекст при десятках установленных скилов.

Агент выбирает нужный скилл **автоматически** по описанию задачи.
Принудительный вызов: упомянуть название скила в промте.

Где искать скилы:
- https://smithery.ai — с категориями и провайдерами (Anthropic, GitHub, Microsoft)
- https://skills.sage — установка командой из сайта

Безопасность: перед установкой проверить что в SKILL.md нет `curl`, `export`, `rm -rf`.

## Правило 7 — 6 паттернов для сложных задач

| Паттерн | Когда использовать |
|---------|-------------------|
| Classify and Act | Поток разнородных входов (письма, заявки) |
| Fan-out + Synthesize | Параллельная проверка нескольких аспектов |
| Adversarial Verification | Критическая задача: один пишет, другой ищет дыры |
| Generate + Filter | Нужно много вариантов → выбрать лучший |
| Tournament | Несколько подходов → выбрать победителя |
| Loop Until Done | Итеративная задача с критериями готовности |

## Правило 8 — Безопасность settings.json

При работе с Bypass Permissions добавить в deny:
- Доступ к `.env` файлам (API-ключи)
- Команды удаления файлов без подтверждения
- Отправка данных на внешние серверы

## Готовые агентные паттерны (референс)

При создании агентных систем использовать как основу:
Router (тимлид) → DeepResearch → Parser → NewsDigest →
DocAnalyzer → YouTube Analyzer → ReportGenerator → MeetingNotes
