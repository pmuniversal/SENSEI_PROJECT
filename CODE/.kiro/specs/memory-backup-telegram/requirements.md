# Requirements Document

Защита памяти — копия в Telegram-канал

## Introduction

Самая ценная вещь в проекте Sensei — это файл памяти `memory.db`. В нём хранятся вся история диалогов, задачи и напоминания. Сейчас бот делает резервные копии (`bot/services/backup.py`) только **на том же сервере** — раз в час копирует базу в папку `data/backups`. Проблема: если сервер (Oracle Cloud VPS) выйдет из строя или будет удалён, то пропадут и база, и все локальные копии одновременно. Всё будет потеряно.

Эта функция добавляет **внешнее (offsite) резервное копирование** — отправку копий в приватный Telegram-канал, где бот уже является администратором. Так копии будут храниться вне сервера, и при гибели сервера их можно будет восстановить.

Что будет делать функция:

1. **Каждые 6 часов** автоматически отправлять в канал: (а) файл базы `memory.db` и (б) текстовый/markdown-экспорт памяти — человекочитаемый файл, который можно дать любому ИИ, чтобы тот «узнал» пользователя.
2. **Раз в неделю** отправлять в канал архив с кодом бота — чтобы при необходимости можно было восстановить и самого бота.
3. Команда **«сделай копию сейчас»** (а также команда `/backup`) — мгновенно отправляет свежие копии в канал по запросу пользователя.

Главный принцип проекта: **Стабильность > Красота > Новые фичи.** Эта функция добавляется как **новый отдельный модуль** и **не изменяет** существующую логику. Локальное часовое копирование (`backup_loop`) сохраняется без изменений — новая функция его дополняет, а не заменяет.

## Glossary

- **Sensei** — Telegram-бот (личный ИИ-наставник), к которому относятся все требования.
- **Telegram_Backup_Service** — новый программный модуль (планируемый файл `bot/services/telegram_backup.py`), отвечающий за подготовку и отправку резервных копий в Telegram-канал.
- **Backup_Loop_Offsite** — новый фоновый процесс (демон-поток), запускаемый из `bot/main.py`, который по расписанию вызывает `Telegram_Backup_Service`.
- **Backup_Channel** — приватный Telegram-канал назначения с идентификатором `-1003978847512`, где бот является администратором. Это адресат всех внешних копий.
- **BACKUP_CHANNEL_ID** — настройка (переменная окружения, читается в `bot/config.py`), хранящая идентификатор `Backup_Channel`. Может быть не задана.
- **BACKUP_ENABLED** — необязательный флаг (переменная окружения), позволяющий полностью отключить отправку копий в `Backup_Channel`, даже если `BACKUP_CHANNEL_ID` задан корректным значением.
- **Memory_DB** — файл базы данных `memory.db` по пути `Config.DB_PATH`, содержащий таблицы `memory`, `tasks`, `reminders`.
- **Memory_Export** — человекочитаемый текстовый файл (формат Markdown), содержащий профиль пользователя (по доступным данным) и историю диалога; предназначен для передачи любому ИИ.
- **Code_Archive** — сжатый архив (`.zip`) с исходным кодом бота (содержимое папки `bot/`), пригодный для восстановления бота.
- **Local_Backup_Loop** — существующий фоновый процесс `backup_loop` из `bot/services/backup.py`, делающий локальные копии базы раз в час. Должен остаться без изменений.
- **On_Demand_Backup** — действие немедленной отправки копий в `Backup_Channel`, запускаемое командой пользователя.
- **Backup_Command** — команда `/backup` и/или текстовая фраза «сделай копию сейчас», запускающая `On_Demand_Backup`.
- **Telegram_File_Limit** — ограничение Telegram Bot API на размер отправляемого через `send_document` файла, равное 50 мегабайтам.
- **Owner** — владелец бота (единственный основной пользователь), не являющийся программистом.

## Requirements

### Requirement 1: Настройка канала для резервных копий

**User Story:** Как владелец бота, я хочу указать идентификатор приватного Telegram-канала в настройках, чтобы бот знал, куда отправлять резервные копии, и чтобы при отсутствии настройки бот продолжал работать без сбоев.

#### Acceptance Criteria

1. WHEN модуль `bot/config.py` инициализируется при запуске бота, THE Sensei SHALL читать значение `BACKUP_CHANNEL_ID` из переменной окружения.
2. WHERE переменная окружения `BACKUP_CHANNEL_ID` после удаления пробельных символов содержит непустое значение длиной от 1 до 64 символов, интерпретируемое как идентификатор Telegram-канала (целое число), THE Sensei SHALL использовать это значение как идентификатор `Backup_Channel` для отправки резервных копий.
3. IF переменная окружения `BACKUP_CHANNEL_ID` отсутствует, равна пустой строке или состоит только из пробельных символов, THEN THE Sensei SHALL запуститься в обычном режиме, записать в журнал сообщение об отключённой отправке в канал и пропустить отправку резервных копий в канал.
4. IF значение `BACKUP_CHANNEL_ID` задано, но некорректно (не интерпретируется как идентификатор Telegram-канала), THEN THE Sensei SHALL запуститься в обычном режиме, записать в журнал предупреждение о некорректной настройке канала и пропустить отправку резервных копий в канал.
5. THE Sensei SHALL хранить значение `BACKUP_CHANNEL_ID` только в переменных окружения (файл `.env`) и SHALL NOT записывать токены и секреты в журнал.
6. WHERE переменная окружения `BACKUP_ENABLED` задана значением `false` (без учёта регистра), THE Sensei SHALL пропускать отправку резервных копий в `Backup_Channel`, даже если `BACKUP_CHANNEL_ID` задан корректным значением.
7. WHERE переменная окружения `BACKUP_ENABLED` не задана или равна пустой строке, THE Sensei SHALL считать отправку резервных копий включённой по умолчанию (при корректно заданном `BACKUP_CHANNEL_ID`).

### Requirement 2: Внешнее копирование базы данных каждые 6 часов

**User Story:** Как владелец бота, я хочу, чтобы копия файла памяти `memory.db` автоматически отправлялась в приватный канал каждые 6 часов, чтобы при гибели сервера данные сохранились вне его.

#### Acceptance Criteria

1. WHILE `BACKUP_CHANNEL_ID` задан непустым значением, THE Backup_Loop_Offsite SHALL отправлять файл `Memory_DB` в `Backup_Channel` с интервалом ровно 6 часов (21600 секунд) между началом соседних плановых отправок.
2. WHEN наступает момент плановой отправки, THE Telegram_Backup_Service SHALL отправлять текущий файл `Memory_DB` методом `send_document` на идентификатор `Backup_Channel`.
3. WHEN Telegram_Backup_Service отправляет файл `Memory_DB`, THE Telegram_Backup_Service SHALL прикреплять к сообщению подпись, содержащую дату и время формирования копии в часовом поясе UTC.
4. IF файла `Memory_DB` не существует по пути `Config.DB_PATH` в момент отправки, THEN THE Telegram_Backup_Service SHALL пропустить отправку этого файла, записать предупреждение в журнал с указанием причины и продолжить работу без остановки потока.
5. IF размер файла `Memory_DB` превышает `Telegram_File_Limit` (50 мегабайт) и отправка завершается ошибкой, THEN THE Telegram_Backup_Service SHALL сохранить исходный файл `Memory_DB` без изменений, записать предупреждение в журнал с указанием причины и продолжить работу без остановки потока.
6. THE Backup_Loop_Offsite SHALL продолжать плановые отправки `Memory_DB` каждые 6 часов (21600 секунд) независимо от причины пропуска или ошибки предыдущей отправки.
7. IF отправка файла `Memory_DB` завершается ошибкой по любой иной причине, отличной от превышения `Telegram_File_Limit` (например, сетевой сбой или недоступность `Backup_Channel`), THEN THE Telegram_Backup_Service SHALL сохранить исходный файл `Memory_DB` без изменений, записать предупреждение в журнал с указанием причины и продолжить работу без остановки потока.
8. IF `BACKUP_CHANNEL_ID` не задан или содержит пустое значение, THEN THE Backup_Loop_Offsite SHALL не выполнять отправку `Memory_DB` в `Backup_Channel` и продолжать работу бота без остановки потока.
9. WHEN завершается любая попытка отправки `Memory_DB` (успехом, пропуском или ошибкой), THE Telegram_Backup_Service SHALL фиксировать однозначный итог попытки (успех / пропуск / ошибка с причиной) в журнале.

### Requirement 3: Текстовый экспорт памяти каждые 6 часов

**User Story:** Как владелец бота, я хочу получать вместе с базой человекочитаемый файл с историей диалога и доступным профилем, чтобы любой ИИ мог по нему «узнать» меня, даже без файла базы данных.

#### Acceptance Criteria

1. WHILE `BACKUP_CHANNEL_ID` задан непустым значением, THE Backup_Loop_Offsite SHALL отправлять `Memory_Export` в `Backup_Channel` с интервалом 6 часов (21600 секунд) вместе с файлом `Memory_DB`.
2. WHEN формируется `Memory_Export`, THE Telegram_Backup_Service SHALL включать в файл историю сообщений из таблицы `memory` в хронологическом порядке (от старых к новым) с указанием роли (`user`/`assistant`), текста и времени.
3. WHEN формируется `Memory_Export`, THE Telegram_Backup_Service SHALL включать в файл доступные данные из таблиц `tasks` и `reminders` как часть профиля пользователя.
4. THE Telegram_Backup_Service SHALL формировать `Memory_Export` в формате Markdown с заголовками-разделами и читаемой структурой.
5. WHEN в базе появляются новые типы данных профиля в будущих версиях, THE Telegram_Backup_Service SHALL формировать `Memory_Export` только из фактически доступных на момент экспорта таблиц без сбоя при отсутствии будущих таблиц.
6. IF в момент формирования `Memory_Export` часть таблиц (например, `tasks` или `reminders`) недоступна, THEN THE Telegram_Backup_Service SHALL продолжить экспорт, используя только доступные таблицы (как минимум `memory`), и отметить в файле отсутствующие разделы.
7. IF при формировании `Memory_Export` возникает ошибка чтения базы данных, THEN THE Telegram_Backup_Service SHALL записать ошибку в журнал, пропустить отправку экспорта целиком в текущем цикле (без отправки частичных данных) и продолжить работу.
8. WHEN отправка `Memory_Export` пропущена из-за ошибки чтения базы данных, THE Backup_Loop_Offsite SHALL повторить попытку формирования и отправки `Memory_Export` в следующем плановом цикле через 6 часов.

### Requirement 4: Еженедельный архив кода бота

**User Story:** Как владелец бота, я хочу раз в неделю получать в канал архив с кодом бота, чтобы при гибели сервера можно было восстановить не только данные, но и самого бота.

#### Acceptance Criteria

1. WHILE `BACKUP_CHANNEL_ID` задан непустым значением, THE Backup_Loop_Offsite SHALL отправлять `Code_Archive` в `Backup_Channel` с интервалом 7 дней (604800 секунд).
2. WHEN наступает момент еженедельной отправки, THE Telegram_Backup_Service SHALL создавать `Code_Archive` в формате `.zip` из содержимого папки кода бота (`bot/`).
3. WHEN `Code_Archive` создан, THE Telegram_Backup_Service SHALL отправлять его методом `send_document` на идентификатор `Backup_Channel` с подписью, содержащей дату создания архива.
4. THE Telegram_Backup_Service SHALL исключать из `Code_Archive` файл `.env`, файлы базы данных и папку с данными `data/`, чтобы не отправлять секреты и временные файлы.
5. IF размер `Code_Archive` превышает `Telegram_File_Limit` (50 мегабайт), THEN THE Telegram_Backup_Service SHALL пропустить отправку архива, записать предупреждение в журнал и продолжить работу.
6. IF запись предупреждения в журнал завершается ошибкой (сбой системы логирования), THEN THE Backup_Loop_Offsite SHALL продолжать плановые еженедельные попытки отправки `Code_Archive` без остановки потока.

### Requirement 5: Копия по запросу (команда «сделай копию сейчас»)

**User Story:** Как владелец бота, я хочу одной командой запустить немедленную отправку свежих копий в канал, чтобы не ждать планового времени перед важными изменениями.

#### Acceptance Criteria

1. WHEN Owner отправляет команду `/backup`, THE Sensei SHALL запускать `On_Demand_Backup`.
2. WHEN Owner отправляет текстовое сообщение «сделай копию сейчас», THE Sensei SHALL запускать `On_Demand_Backup`.
3. WHEN запускается `On_Demand_Backup`, THE Sensei SHALL немедленно отправлять Owner-у подтверждающее сообщение «📤 Отправляю копию...».
4. WHEN `On_Demand_Backup` выполнен, THE Telegram_Backup_Service SHALL отправлять в `Backup_Channel` текущий файл `Memory_DB` и `Memory_Export`.
5. WHEN отправка в рамках `On_Demand_Backup` завершена успешно, THE Sensei SHALL отправлять Owner-у сообщение «✅ Копия отправлена в канал».
6. IF `BACKUP_CHANNEL_ID` не задан в момент вызова `Backup_Command`, THEN THE Sensei SHALL отправлять Owner-у сообщение о том, что канал для копий не настроен, и SHALL NOT прерывать работу бота.
7. IF при выполнении `On_Demand_Backup` возникает ошибка отправки, THEN THE Sensei SHALL отправлять Owner-у понятное сообщение об ошибке и записывать ошибку в журнал.

### Requirement 6: Устойчивость к ошибкам и плавная деградация

**User Story:** Как владелец бота, я хочу, чтобы любые сбои отправки копий (сеть, ошибки Telegram, большой файл) никогда не роняли бота и его фоновые процессы, чтобы основная функциональность всегда оставалась доступной.

#### Acceptance Criteria

1. IF при отправке копии в `Backup_Channel` возникает ошибка сети или ошибка Telegram API, THEN THE Backup_Loop_Offsite SHALL перехватывать ошибку, записывать её в журнал и продолжать работу без остановки потока.
2. WHEN обработка планового цикла завершена (успешно или с ошибкой), THE Backup_Loop_Offsite SHALL уходить в ожидание (`sleep`) до следующего планового момента, не нагружая процессор.
3. IF при запуске бота `BACKUP_CHANNEL_ID` отсутствует или некорректен, THEN THE Sensei SHALL запуститься штатно и SHALL NOT прерывать запуск из-за отсутствия настройки копий.
4. THE Backup_Loop_Offsite SHALL работать как поток-демон, который автоматически завершается вместе с остановкой бота.
5. THE Telegram_Backup_Service SHALL записывать в журнал имя отправленного файла и итог операции (успех/пропуск/ошибка) без раскрытия токенов и секретов.

### Requirement 7: Сохранение существующей функциональности

**User Story:** Как владелец бота, я хочу быть уверен, что добавление внешнего копирования не сломает текущую работу бота, чтобы переписка, задачи, напоминания и голос продолжали работать как прежде.

#### Acceptance Criteria

1. THE Sensei SHALL сохранять без изменений обработку текстовых сообщений, голосовых сообщений и команд `/task`, `/tasks`, `/remind`.
2. THE Sensei SHALL сохранять работу существующего `Local_Backup_Loop` (часовое локальное копирование в `data/backups`) без изменений.
3. THE Sensei SHALL сохранять работу существующего цикла напоминаний `reminder_loop` без изменений.
4. THE Telegram_Backup_Service SHALL быть реализован как новый отдельный модуль и SHALL NOT изменять существующие модули `ai.py`, `memory.py`, `tasks.py`, `reminders.py`, `backup.py`, `db.py`.
5. WHEN запускается `bot/main.py`, THE Sensei SHALL запускать `Backup_Loop_Offsite` дополнительно к существующим фоновым процессам, не заменяя их.
6. THE Backup_Command SHALL распознаваться точно и SHALL NOT перехватывать или изменять поведение существующих команд `/task`, `/tasks`, `/remind`.

### Requirement 8: Надёжность расписания при перезапусках

**User Story:** Как владелец бота, я хочу, чтобы копии действительно создавались регулярно даже после перезапуска контейнера или сервера, чтобы длительные простои не оставляли меня без свежих внешних копий.

#### Acceptance Criteria

1. WHEN бот запускается, THE Backup_Loop_Offsite SHALL начинать отсчёт интервалов плановых отправок от момента старта процесса.
2. WHERE включён режим устойчивого расписания (хранение времени последней отправки), THE Telegram_Backup_Service SHALL сохранять время последней успешной отправки каждого типа копии (база, экспорт, архив кода) в постоянном хранилище.
3. WHERE включён режим устойчивого расписания, WHEN бот запускается после простоя, превышающего плановый интервал какого-либо типа копии, THE Backup_Loop_Offsite SHALL выполнять отправку этого типа копии при первой возможности после старта.
4. THE Backup_Loop_Offsite SHALL гарантировать отправку каждого типа копии не реже одного раза за его плановый интервал при условии работающего бота и доступного канала.

## Открытые вопросы и точки решения (для фазы проектирования)

> Эти пункты не являются требованиями, но фиксируют решения, которые нужно принять при проектировании.

1. **Способ соблюдения расписания.** Требование 8 описывает два варианта: (а) простые интервальные циклы от момента старта процесса (проще, но при частых перезапусках момент отправки «плавает»); (б) хранение времени последней отправки в постоянном файле/таблице (надёжнее переживает простои). Нужно выбрать вариант на этапе проектирования. Владельцу важна сама регулярность копий, а не точность по часам.
2. **Имя модуля.** Предлагается `bot/services/telegram_backup.py` (либо `cloud_backup.py`). Выбрать окончательно при проектировании.
3. **Где регистрируется команда `/backup`.** Предлагается добавить в `bot/handlers/commands.py` обработчик команды `/backup` и распознавание фразы «сделай копию сейчас» (последнее — в `text.py` либо отдельным обработчиком), сохранив точное сопоставление команд.
4. **Состав `Memory_Export`.** Полноценный «личный профиль» — это более поздний этап дорожной карты (Этап 7). Сейчас экспорт строится из доступных таблиц (`memory`, `tasks`, `reminders`) и должен быть совместим с будущими расширениями.
