## 02.06.2026 — Модульная архитектура + перенос токенов в .env

**Что изменили:**
- Разбили монолитный `main.py` (~450 строк) на модульную структуру в папке `bot/`
  (handlers / services / database / utils).
- Перенесли секретные токены (TELEGRAM_TOKEN, OPENAI_API_KEY) из кода в файл `.env`.
- Добавили чтение `.env` через библиотеку `python-dotenv`.
- Данные (база, документы, голос, бэкапы) теперь хранятся в папке `data/`.
- Исправили скрытый баг: команда `/tasks` раньше ошибочно работала как `/task`
  (создавала задачу вместо показа списка). Теперь команды распознаются точно.

  (запуск через Supervisor не меняется).

**Почему изменили:**
- Безопасность: токены больше не лежат открыто в коде.
- Масштабируемость: новые функции можно добавлять отдельными модулями,
  не трогая работающий код.
- Стабильность: вся логика сохранена «один в один».

**Какие файлы затронуты:**
- Изменён: `main.py` (теперь переходник)
- Изменён: `requirements.txt` (добавлен `python-dotenv`)
- Созданы: `.env`, `.env.example`, `.gitignore`, `STRUCTURE.md`, `MIGRATION_CHECKLIST.md`
- Созданы: `bot/` со всеми модулями (config, main, handlers/, services/, database/, utils/)
- Созданы папки: `data/` (memory.db, backups, docs, voice)
- Резервная копия: `backups/main.py.bak` (старый рабочий main.py)

**Что проверить:**
- См. `MIGRATION_CHECKLIST.md`, ЧАСТЬ 6 — чек-лист проверки в Telegram.
- Главное: бот отвечает, работают `/task`, `/tasks`, `/remind`, голос, напоминания.

**Откат:**
- `cp /app/main.py.bak /app/main.py` + перезапуск контейнера.
- Подробности отката: `MIGRATION_CHECKLIST.md`, ЧАСТЬ 7.

## 03.06.2026 — Развёртывание новой версии на сервере (VPS) ✅ УСПЕШНО

**Что сделали:**
- Перенесли новую модульную версию кода в контейнер `simplerllm` на VPS (Oracle Cloud).
- Код передавали через консоль Portainer в виде сжатого архива (base64),
  по частям, с проверкой контрольной суммы MD5 (`788dce396d4e72a72d72b6ada6f56954`).
- Распаковали архив в `/app` — папка `bot/` со всеми модулями развернулась.
- Создали на сервере файл `/app/.env` с токенами и параметром `SENSEI_DATA_DIR=/app`.
- Параметр `SENSEI_DATA_DIR=/app` оставляет рабочую базу `memory.db` на старом месте
  (`/app/memory.db`) — данные НЕ переносились и НЕ потеряны.
- Переписали `/etc/supervisord.conf`: добавили секции `[unix_http_server]`,
  `[supervisorctl]`, `[rpcinterface]`, параметры `directory=/app`, `startretries=999`,
  `startsecs=5` — чтобы Supervisor стабильно держал бота и не «сдавался» после сбоев.
- Старый конфиг сохранён: `/etc/supervisord.conf.bak`.

**Проблемы по пути и решения:**
- Веб-консоль Portainer обрезала/искажала длинные вставки → перешли на мелкие блоки
  по 15 строк + проверка MD5; битые строки находили делением пополам и заменяли точечно.
- Команды `supervisorctl` не работали (в конфиге не было нужных секций) → добавили их.
- Бот падал с `ConnectionError: Connection aborted` → причина: несколько копий бота
  с одним токеном одновременно конфликтовали за Telegram. После запуска ОДНОЙ копии
  через Supervisor — связь стабильна.

**Проверки (пройдены):**
- `curl .../getMe` → `"ok":true`, бот `Bekhzod_2_bot` (id 8459988161) — токен живой.
- Код: `CONFIG_OK` + `IMPORTS_OK` — модули исправны, токены читаются.
- Лог: `AI OS STARTED`.
- Бот отвечает в Telegram. ✅

**Текущее состояние на сервере:**
- `/app/bot/` — новый модульный код
- `/app/main.py` — переходник
- `/app/.env` — токены + SENSEI_DATA_DIR=/app
- `/app/memory.db` — рабочая база (на старом месте, данные целы)
- `/app/main.py.bak`, `/app/memory.db.bak` — резервные копии для отката
- `/etc/supervisord.conf` — обновлённый автозапуск (бэкап: .bak)

**Откат (если понадобится):**
- `cp /app/main.py.bak /app/main.py` + `cp /etc/supervisord.conf.bak /etc/supervisord.conf`

**Известные мелочи (не критично):**
